# IOM Module Contract — React Frontend

## 1. Purpose

This document defines the frontend implementation contract for the **IOM — Internal Office Memo** module in the Puravankara Portal.

The goal is to give React developers and AI coding agents a stable, implementation-ready reference before backend APIs are fully finalized. For now, API integration can be implemented with mock responses while preserving the expected frontend contracts, status model, role-wise actions, list columns, validation rules, and open dependencies.

---

## 2. Frontend Scope

The IOM module will be implemented inside the existing React frontend architecture.

### Existing frontend rules to follow

- Use existing role-based routing under `src/routes/sections/*-routes.tsx`.
- Use existing path constants from `src/routes/paths.ts`.
- Use MUI-first UI patterns.
- Use MUI X DataGrid for record lists.
- Use Redux Toolkit for module state.
- Use existing Axios wrappers from `src/services/`.
- Use centralized API route constants.
- Do not create a new top-level `src/*` bucket.
- Prefer colocating UI under `src/sections/common-module/iom/` because the module is shared across CRM, Finance, Loyalty, and Super Admin views.
- Mock APIs are allowed initially, but the service interface should match the expected backend contract.

---

## 3. Source Requirements Summary

The Excel requirement defines the following high-level capabilities:

1. CRM, Loyalty, and Finance users log in through existing SSO + OTP.
2. CRM users upload signatures from profile management.
3. CRM sees only project-specific 15% qualified booking records from SAP where source is Loyalty.
4. CRM generates IOM from prefilled booking data.
5. CRM submits IOM for verifier approval.
6. CRM Verifier/TL verifies IOM and places signature.
7. CRM Head approves IOM and places signature.
8. Finance receives CRM Head approved IOM records.
9. Finance places signature and closes the IOM.
10. Loyalty receives Finance closed records.
11. Loyalty fetches/matches Pinelabs customer details.
12. Loyalty creates a new customer profile if not found.
13. Loyalty marks points as eligible/redeemable and releases loyalty points.
14. Loyalty requests invoice in bulk.
15. Loyalty submits invoice details in bulk.
16. Finance can close/update invoice.
17. Users can review generated IOM as PDF.
18. Failed email notifications should expose a retry CTA.
19. Super Admin can view all IOM records.

## 4. Roles Involved

| Role | Purpose in IOM Module | Frontend Access |
|---|---|---|
| CRM User / Created By | Views eligible booking records, uploads signature, generates IOM, submits for approval. | CRM IOM list, Generate IOM, IOM detail/PDF review. |
| CRM Verifier / CRM TL | Verifies IOM generated by CRM and places verifier signature. | Verification queue/detail view. |
| CRM Head | Approves verified IOM and places approval signature. | Approval queue/detail view. |
| Finance User | Views CRM Head approved records, signs and closes IOM, closes/updates invoice. | Finance IOM list/detail, Finance closure, Invoice closure. |
| Loyalty User | Views Finance closed records, handles Pinelabs matching/profile creation, points release, invoice request/submission, daily pool. | Loyalty IOM list/detail, Pinelabs panel, Points panel, Invoice bulk actions, Daily pool. |
| Super Admin | Views all IOM records for monitoring/audit. Metrics are out of scope for now. | Read-only all-records view. |

### Role assumptions

- Exact backend role constants must be mapped against existing `ROLES` in `src/utils/constant.ts`.
- If CRM Verifier/TL and CRM Head are not separate role constants, frontend should rely on backend-provided `allowedActions` rather than only local role checks.
- Super Admin should not get operational actions unless explicitly approved later.

## 7. Allowed Actions Per Role

Frontend should prefer `allowedActions` from API response. Local role/status checks should be used only as a secondary guard for UI rendering.

| Role | Allowed Actions |
|---|---|
| CRM User / Created By | `ViewRecord`, `UploadSignature`, `GenerateIom`, `SubmitForApproval`, `ReviewPdf`, `RetryNotification`, `ExportRecords` |
| CRM Verifier / CRM TL | `ViewRecord`, `VerifyIom`, `ReviewPdf`, `RetryNotification`, `ExportRecords` |
| CRM Head | `ViewRecord`, `ApproveIom`, `ReviewPdf`, `RetryNotification`, `ExportRecords` |
| Finance User | `ViewRecord`, `CloseIomFinance`, `CloseInvoice`, `ReviewPdf`, `RetryNotification`, `ExportRecords` |
| Loyalty User | `ViewRecord`, `FetchPinelabsCustomer`, `MatchPinelabsCustomer`, `CreatePinelabsProfile`, `MarkPointsEligible`, `MarkPointsRedeemable`, `ReleasePoints`, `RequestInvoiceBulk`, `SubmitInvoiceBulk`, `ReviewPdf`, `RetryNotification`, `ExportRecords` |
| Super Admin | `ViewRecord`, `ReviewPdf`, `ExportRecords` |

### Important action rules

1. CRM cannot generate IOM unless CRM signature exists.
2. Only one active/approved IOM should be allowed per booking record.
3. CRM Verifier can verify only after CRM submission.
4. CRM Head can approve only after CRM verification.
5. Finance can close IOM only after CRM Head approval.
6. Loyalty can process points only after Finance closure.
7. Loyalty can create Pinelabs profile only when no customer match exists.
8. Invoice request should support bulk selection.
9. Invoice submission should support bulk update.
10. Notification retry should be visible only when notification failure is returned by API.

## 10. Signature Upload Rules

### 10.1 Applicable users

Signature upload is required for any user who must place a signature on IOM:

- CRM User / Created By
- CRM Verifier / CRM TL
- CRM Head
- Finance User

The Excel specifically calls out CRM profile signature upload. Frontend should keep the component reusable so it can support verifier/head/finance signature upload if backend requires signatures from their profile too.

### 10.2 File validation

| Rule | Value |
|---|---|
| Allowed extensions | `.jpg`, `.jpeg`, `.png` |
| Allowed MIME types | `image/jpeg`, `image/jpg`, `image/png` |
| Max file size | `10 MB` |
| Required before Generate IOM | Yes, for CRM user |
| Preview before upload | Yes |
| Replace existing signature | Allowed only if backend permits |
| OCR/background crop | Backend dependency; frontend uploads image and shows processed URL if returned |

### 10.3 CRM generate IOM guard

## 11. PDF Review Requirement

### Requirement

Generated IOM must be viewable in PDF format after submission.

### Frontend behavior

1. Show **Review IOM** / **View PDF** action after IOM is generated or submitted.
2. PDF should be available in detail page and optionally in a modal/drawer.
3. PDF should refresh after each signature/approval step if backend generates a new version.
4. UI should show PDF generation status:
   - `Not Generated`
   - `Generating`
   - `Generated`
   - `Generation Failed`
5. If PDF generation fails, show retry only if backend provides retry capability.
6. The viewer should support:
   - Open in modal/drawer
   - Download, if allowed
   - Open in new tab, if URL-based
7. PDF URL/blob/base64 format should be abstracted inside service/helper.

### Preferred response shape

```ts
export interface IomPdfResponse {
  iomId: string;
  pdfStatus: 'NOT_GENERATED' | 'GENERATING' | 'GENERATED' | 'FAILED';
  pdfUrl?: string;
  pdfBase64?: string;
  generatedAt?: string;
  errorMessage?: string;
}
```

## 12. Notification Retry Rule

### Notification flow

1. CRM generates/submits IOM → email to CRM Verifier.
2. CRM Verifier verifies IOM → email to CRM Head.
3. CRM Head approves IOM → email to Finance.
4. Finance closes IOM → email to Loyalty.
5. Loyalty requests invoice → notification to Pinelabs.

### Retry CTA visibility

Show **Retry Notification** only when API response includes failed notification state.

```ts
export interface IomNotificationState {
  step: IomWorkflowStep;
  status: 'NOT_REQUIRED' | 'PENDING' | 'SENT' | 'FAILED';
  lastAttemptAt?: string;
  failureReason?: string;
  retryAllowed: boolean;
}
```

CTA rule:

```ts
const canRetryNotification =
  notification.status === 'FAILED' &&
  notification.retryAllowed &&
  record.allowedActions.includes(IomAction.RetryNotification);
```

### Frontend constraints

- Do not auto-retry repeatedly from frontend.
- Do not expose retry CTA if failure is not confirmed by backend.
- After retry, refresh detail/list row state.
- Show user-friendly success/failure toast.
- Keep notification failure visible in audit/history panel.

## 16. Frontend Validation Rules

### 16.1 List fetch rules

- CRM sees only assigned project records.
- Super Admin sees all records.
- Finance sees records approved by CRM Head.
- Loyalty sees records closed/approved by Finance.
- Filters and pagination should be server-driven from the beginning.

### 16.2 IOM generation rules

- Booking source must be Loyalty source:
  - Purva Privilege
  - Provident Premier
- 15% payment qualification must be achieved.
- CRM signature must exist.
- Only one active/approved IOM per booking record.
- Dynamic brand/entity must be derived from project/brand.
- SAP-driven fields should be read-only unless backend explicitly marks them editable.
- If payment fields are editable, show clear warning that changes will not be pushed back to SAP.

### 16.3 Approval rules

- CRM submits generated IOM to verifier.
- CRM Verifier signs and verifies.
- CRM Head signs and approves.
- Finance signs and closes.
- Loyalty acts only after Finance closure.

### 16.4 Loyalty rules

- Fetch Pinelabs customer before releasing points.
- If customer exists, match/confirm before points release.
- If customer does not exist, create profile using email and phone number.
- Release loyalty points only after customer is matched or created.
- Push data to Salesforce only if backend endpoint confirms support.

### 16.5 Invoice rules

- Invoice request supports bulk selection.
- Invoice submission supports bulk update.
- Invoice submission requires:
  - Invoice number
  - Invoice submitted date
  - Amount with GST
- Finance can close/update invoice after submission.

## 19. Acceptance Criteria

### CRM

- CRM can see eligible project-specific 15% qualified loyalty booking records.
- CRM cannot generate IOM without uploaded signature.
- CRM can upload valid signature file from profile.
- CRM can generate IOM from prefilled booking data.
- CRM can submit generated IOM for verification.
- CRM can review generated IOM PDF after submission.

### CRM Verifier / CRM Head

- CRM Verifier can see submitted IOM records pending verification.
- CRM Verifier can verify and sign IOM.
- CRM Head can see verified IOM records pending approval.
- CRM Head can approve and sign IOM.
- Approval trail is visible on detail view.

### Finance

- Finance can see CRM Head approved IOM records.
- Finance can review PDF.
- Finance can sign and close IOM.
- Finance closure moves record to Loyalty queue.
- Finance can close/update invoice where allowed.

### Loyalty

- Loyalty can see Finance closed records.
- Loyalty can fetch Pinelabs customer details.
- Loyalty can match customer or create new profile when not found.
- Loyalty can mark points as eligible/redeemable/released.
- Loyalty can request invoices in bulk.
- Loyalty can submit invoice details in bulk.
- Loyalty can view daily pool amount when API is available.

### Super Admin

- Super Admin can view all records.
- Super Admin cannot perform operational actions in current scope.

### Shared

- All list views support loading, empty, error, and pagination states.
- Actions are driven by `allowedActions` and status.
- Notification retry CTA appears only for failed notification state.
- PDF preview handles loading and failure states.
- Mock APIs can be swapped with real APIs without changing UI contracts.
